# 为什么继续补复杂节点，以及怎么避免 skill 变乱

## 结论先说

继续补复杂节点 **有意义**，但前提是：
- 不是为了“支持更多名词”
- 而是为了覆盖真实高频业务
- 并且必须通过模板、路由、检查单来约束使用路径

如果只是一味堆节点知识，不做行动指导，skill 会越来越乱。

---

## 继续补复杂节点的意义

### 1. 因为真实业务根本不止 start → llm → answer
在真实 Dify 项目里，常见需求天然需要：
- `if-else`：分支判断
- `tool`：工具调用
- `http-request`：外部 API
- `code`：数据整理
- `iteration`：批量处理 / 循环收集
- `variable-aggregator`：多路结果汇总
- `question-classifier`：分类路由

如果 skill 只会最简骨架，那它只能做 demo，不能做业务。

### 2. 复杂节点补齐后，skill 才能真正“指导行动”
比如：
- 客服模板应该自然推荐 `if-else + tool`
- API 编排自然推荐 `parameter-extractor + http-request + code`
- RAG 自然推荐 retrieval + answer

这时候 skill 不是在炫技，而是在给用户最合适的骨架。

### 3. 审查和排障也必须理解复杂节点
如果 skill 不懂这些节点，它就没法：
- 发现结构错误
- 给最小修复建议
- 识别导入和运行风险

---

## 风险：会不会越补越乱？

会。
如果用错方法，skill 会出现 3 个典型问题：

### 风险 1：知识越来越多，但用户不会用
解决方法：
- 永远先路由任务类型
- 永远先选模板
- 不让用户直接面对全部节点表

### 风险 2：会生成“理论上很强、实际上跑不通”的 DSL
解决方法：
- 新节点加入前必须至少有一个最小模板
- 模板要能被 review / validate
- 高风险节点默认标注“需人工复核”

### 风险 3：用户提一个简单需求，却被复杂输出淹没
解决方法：
- 简单任务仍走简单模板
- 复杂节点按需升级，而不是默认全开

---

## 正确原则：只补“高频且可行动”的复杂节点

优先补这些：
1. `iteration`
2. `variable-aggregator`
3. `question-classifier`
4. `knowledge-retrieval`
5. `assigner / conversation variable`

原因：
- 真实业务高频
- 能形成稳定模板
- 能配套审查规则

不应优先补那些：
- 很少见且高度依赖特定版本内部实现的节点
- 没有稳定模板、没法验证、没法给排障路径的节点

---

## 如何保证“有效性”

判断一个新增节点值不值得补，只看 4 条：

1. **是否有真实业务场景高频使用**
2. **是否能形成模板**
3. **是否能形成审查规则**
4. **是否能形成排障动作**

四条至少满足三条，才值得纳入 skill。

---

## 对外推广时怎么讲这件事

不要说：
- “这个 skill 支持很多很多节点，很牛”

要说：
- “这个 skill 会根据业务场景，自动选择合适的 Dify 骨架与节点组合”
- “简单需求走简单模板，复杂需求才启用复杂节点”
- “它不只是生成，还能审查、修复、校验”

这才是有效性。
