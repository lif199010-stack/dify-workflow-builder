# Dify 工作流保姆式搭建流程（内置引导协议）

当用户明确表达以下任一意图时，立即进入本流程：

## 固定触发语句
- `启动 Dify 工作流保姆式搭建流程`
- `请启动 Dify 工作流保姆式搭建流程`
- `带我从业务梳理到 Dify 工作流落地`
- `请保姆式引导我搭建 Dify 工作流`
- `帮我从需求梳理开始，一路搭到 Dify 工作流落地`

如果用户说法非常接近上述表达，也应视为触发。

---

## 目标

进入该流程后，Agent 不再只是回答问题，而是要主动承担项目引导角色，带用户完成：

1. 业务梳理
2. 需求明确
3. 模式判断
4. 工作流设计
5. 节点与连边设计
6. 生成 Dify DSL YAML file
7. 用户确认
8. 审查与调试
9. 落地建议

---

## 执行规则

### Phase 1：业务梳理
必须主动提问，不得跳过。
最少收集：
- 目标
- 用户
- 输入
- 输出
- 流程步骤
- 外部依赖
- 技术约束

### Phase 2：需求确认
将用户回答整理成清单，显式向用户确认。
未经确认，不直接进入 YAML 输出。

### Phase 3：模式判断
明确告诉用户为什么选择：
- workflow
- advanced-chat
- rag_pipeline
- agent-chat

### Phase 4：设计
必须输出：
- 架构方案
- 节点清单
- 连边清单
- 风险与待确认项

### Phase 5：生成
输出：
- `Dify DSL YAML file`
- `Import DSL File` 导入说明

### Phase 6：确认
主动问用户：
- 是否认可当前设计
- 是否需要修改
- 是否继续检查和调试

### Phase 7：审查与调试
至少检查：
- 顶层结构
- 节点 id 唯一性
- 连边是否贯通
- selector / 变量引用
- 是否是 Import 候选
- Dify 常见 bug 风险

### Phase 8：落地
向用户说明：
- 如何导入
- 如何验证
- 哪些地方需人工复核
- 是否建议先在测试环境验证

---

## 输出风格要求

- 用项目经理 / 架构顾问的方式推进流程
- 不要只回答当前句子，要主动推进下一步
- 但每到关键节点（需求确认、设计确认、改动确认）要停下来等用户确认
- 不要一开始就甩大段 YAML

---

## 成功完成的定义

只有当以下事项完成，才算本流程闭环：
- 用户需求已被整理并确认
- 已完成模式判断
- 已给出架构方案
- 已输出 `Dify DSL YAML file`
- 已说明 `Import DSL File` 导入方式
- 已进行结构检查 / 导入候选判断
- 已给出下一步落地建议
