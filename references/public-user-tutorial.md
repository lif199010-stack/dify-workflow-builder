# dify-workflow-builder 使用教程（面向 Dify 使用者）

这是一份可以直接对外发布的教程稿，面向第一次接触 `dify-workflow-builder` 的 Dify 使用者。

---

# 一、这是什么

`dify-workflow-builder` 是一个专门帮助你**搭建 Dify 工作流**的 Agent Skill。

你可以把它理解成：

- 一个会做 **业务梳理** 的顾问
- 一个会做 **工作流设计** 的架构师
- 一个会输出 **Dify DSL YAML file** 的生成器
- 一个会帮你做 **审查、修复、校验、调试** 的助手

它不是只会写 YAML，真正强的地方是：

**它能把你的业务需求，一步一步带到 Dify 工作流落地。**

---

# 二、它能做什么

`dify-workflow-builder` 主要可以做 5 类事：

## 1. 从业务需求生成 Dify 工作流
比如：
- 生成智能客服工作流
- 生成 OCR 图片识别工作流
- 生成 API 编排工作流
- 生成知识库问答工作流
- 生成 Agent 工具调用工作流

## 2. 输出可导入的 Dify DSL YAML file
它会生成完整的：
- `app`
- `dependencies`
- `kind`
- `version`
- `workflow`

并告诉你如何通过 Dify 的 **Import DSL File** 导入。

## 3. 审查现有的 Dify YAML / DSL
比如：
- 这个 YAML 能不能导入？
- 结构有没有问题？
- 哪个节点配置有风险？

## 4. 修复和重构已有工作流
比如：
- 最小修复导入失败的 DSL
- 重构混乱的工作流结构
- 把简单 workflow 升级为 rag_pipeline / agent-chat

## 5. 做保姆式引导搭建
也就是：
- 先帮你梳理业务
- 再帮你明确需求
- 再帮你设计工作流
- 再生成 YAML
- 再检查和调试
- 最后帮助你落地到 Dify

---

# 三、你在哪里用这个 skill

你是在 **带有该 skill 的 Agent** 里使用它，不是去 Dify 里面直接点一个按钮。

也就是说：

1. 先打开已经安装了 `dify-workflow-builder` 的 Agent
2. 在聊天框里输入你的需求
3. Agent 识别到这是 Dify 工作流任务后，就会调用这个 skill
4. 然后开始引导你完成工作流搭建

---

# 四、怎么触发这个 skill

你可以直接说下面这些话，来触发 Agent 调用 `dify-workflow-builder`：

## 最通用触发语句

```text
帮我搭一个 Dify 工作流
```

## 更明确的触发语句

```text
帮我根据业务需求生成 Dify DSL YAML file
```

```text
帮我设计并搭建一个 Dify workflow
```

```text
帮我 review 这个 Dify YAML
```

```text
帮我 refactor 这个 Dify DSL
```

```text
帮我检查这个 Dify YAML 能不能通过 Import DSL File 导入
```

---

# 五、最推荐的固定语句（建议对外教学时直接教这个）

如果你想让 Agent 直接进入**保姆式全流程引导**，推荐使用下面这句固定语句：

```text
启动 Dify 工作流保姆式搭建流程
```

或者更完整一点：

```text
请启动 Dify 工作流保姆式搭建流程，带我从业务梳理到需求明确，再到工作流设计、搭建、检查、调试，直到可以落地。
```

这句话的作用是：

让 Agent 不再只是“回答你”，而是自动进入一个完整流程：

1. 业务梳理
2. 需求确认
3. 模式判断
4. 架构设计
5. 节点规划
6. 生成 Dify DSL YAML file
7. 用户确认
8. 检查与调试
9. 落地建议

# 六、Agent 调用 skill 后会怎么带你走

当你输入固定语句后，Agent 应该这样工作：

## 第 1 步：业务梳理
Agent 会先问你：
- 这个工作流解决什么问题？
- 目标用户是谁？
- 输入是什么？
- 输出是什么？
- 中间流程有哪些步骤？
- 需要接什么 API / 知识库 / 工具？

## 第 2 步：需求明确
Agent 会把你的回答整理成需求确认单，例如：
- 目标
- 用户
- 输入
- 输出
- 关键流程
- 外部依赖
- 技术约束

然后让你确认。

## 第 3 步：模式判断
Agent 会判断当前更适合：
- `workflow`
- `advanced-chat`
- `rag_pipeline`
- `agent-chat`

并告诉你为什么这么判断。

## 第 4 步：工作流设计
接着 Agent 会输出：
- 架构方案
- 节点清单
- 连边清单
- 关键变量 / selector
- 风险与待确认项

## 第 5 步：生成 YAML
然后 Agent 会输出：
- **Dify DSL YAML file**
- 并明确说明如何通过 **Import DSL File** 导入

## 第 6 步：用户确认
Agent 会请你确认：
- 是否接受这份工作流设计
- 是否继续调整
- 是否继续进入检查 / 调试 / 落地阶段

## 第 7 步：检查和调试
Agent 会检查：
- 顶层结构
- 节点配置
- 连边关系
- 变量引用
- Import 候选状态
- 常见 Dify bug 风险

## 第 8 步：落地建议
最后 Agent 会告诉你：
- 如何导入
- 如何调试
- 哪些地方需要人工复核
- 哪些地方建议先在测试环境验证

# 七、你应该怎么跟 Agent 说

## 场景 1：我什么都没有，只知道业务目标
你可以这样说：

```text
启动 Dify 工作流保姆式搭建流程。
我要做一个工作流：用户上传图片后，自动识别文字并返回结构化结果。
```

## 场景 2：我已经知道想做什么模式

```text
帮我搭一个 workflow。
目标：用户提交表单后，调用 API 生成结果并返回。
```

## 场景 3：我想让它生成可导入 YAML

```text
根据下面业务需求，输出完整的 Dify DSL YAML file，并告诉我如何用 Import DSL File 导入。
```

## 场景 4：我已经有 YAML，只想检查

```text
帮我检查这个 Dify YAML 能不能导入，并列出阻塞项。
```

## 场景 5：我已经有 YAML，想修

```text
帮我最小修复这个 Dify YAML，修完后告诉我是否可以导入。
```

# 八、标准输出应该长什么样

正常情况下，Agent 不应该一上来就丢给你一大段 YAML。

更合理的顺序应该是：

1. 任务概述
2. 模式判断
3. 架构方案
4. 节点清单
5. 连边清单
6. 风险与待确认项
7. `Dify DSL YAML file`
8. `Import DSL File` 导入说明

如果只给 YAML，没有前面的分析和确认，通常不够稳。

---

# 九、怎么在 Dify 里真正使用生成结果

当 Agent 输出 `Dify DSL YAML file` 后，你要这样做：

## 步骤 1
把 YAML 内容保存成一个文件，例如：

- `my-workflow.yml`
- `my-workflow.yaml`

## 步骤 2
打开 Dify 控制台

## 步骤 3
找到：

**Import DSL File**

## 步骤 4
上传刚才保存的 YAML 文件

## 步骤 5
导入后，进入工作流画布检查：
- 节点是否完整
- 工具是否绑定正确
- 模型是否可用
- 文件上传 / 变量 / selector 是否正常

## 步骤 6
运行测试，确认业务逻辑真的跑通

# 十、常见误区

## 误区 1：一句话让 Agent 直接生成复杂工作流
问题：上下文太少，工作流容易偏。

正确方式：至少说清楚目标、输入、输出、流程。

## 误区 2：导入成功就等于搭建成功
问题：Dify 的导入成功，只能说明结构层面没完全阻塞。

正确方式：导入后还要跑真实测试。

## 误区 3：把所有节点都加进去才叫高级
问题：工作流不是节点越多越好。

正确方式：让 Agent 根据业务选合适骨架。

## 误区 4：遇到问题就大改 YAML
问题：很多问题其实是 Dify 的版本、插件、导入机制或 UI bug。

正确方式：先 review，再 refactor，再 validate。
